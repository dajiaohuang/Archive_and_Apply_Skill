# 条目模板

这个模板用于维护 `experiences/`、`projects/`、`publications/` 下的源条目。原则是先把事实写全，再从这里提炼简历、面试稿和定制材料。

## 使用原则

- 默认用中文写，英文信息只在必要处保留原名
- 先写“源事实”，不要一开始就压缩成简历 bullet
- 能量化就量化，不能确定就先空着，不要编
- 项目、经历、论文共用一套大结构，但允许按类型增删字段

## 推荐文件头

标题直接使用条目名称，例如：

```md
# 条目标题示例
```

## 通用模板

```md
# 条目名称

## 基本信息

- 类型：实习 / 科研实习 / 项目 / 论文 / 竞赛 / 其他
- 时间：
- 角色：
- 团队 / 机构：
- 导师 / 合作者：
- 方向：
- 相关链接：
  - 

## 一句话概述

用 1 到 2 句话说明：这是一个什么场景，我解决了什么问题，我主要做了什么。

## 背景与目标

说明这个条目的背景、目标用户或研究问题，以及为什么值得做。

## 我的工作

按模块展开，不要只写抽象职责，尽量写清：

- 我实际负责了什么
- 我的边界在哪里
- 我交付了哪些具体模块、工具、流程或实验

如果内容较多，推荐像下面这样分小节：

### 1) 模块 A

- 

### 2) 模块 B

- 

## 技术方案

写清楚核心技术路径，例如：

- 用了哪些模型、框架、系统、工具链
- 数据如何组织、清洗、训练、评测、部署
- 关键工程设计、架构权衡、接口约束、风险控制

## 结果与产出

优先写硬结果：

- 指标、排名、论文、上线效果、测试覆盖、用户价值、可交付物
- 如果是研究，写 acceptance / benchmark / challenge / 数据规模
- 如果是工程，写系统能力、接口、稳定性、性能、风险边界

## 难点与复盘

说明：

- 最难的问题是什么
- 你是怎么处理的
- 这件事带来了什么可复用经验

## 技术关键词

`Keyword1` `Keyword2` `Keyword3`

## 后续可补充

- [ ] 数据规模 / 用户规模 / 文档规模
- [ ] 定量指标
- [ ] 我的具体边界
- [ ] 架构图 / 流程图 / 系统图
- [ ] 面试里可展开的问题
```

## 按类型补充字段

### 1. `experiences/` 适用

适合实习、科研实习、实验室经历。通常建议补这些字段：

```md
- 导师：
- 团队 / 实验室：
- 方向：
```

写法重点：

- 强调你参与的完整链路，而不只是“协助”
- 研究型经历可把“数据处理 / 模型验证 / 渲染可视化 / benchmark 结果”拆开写
- 如果和论文、比赛直接相关，最好在“结果与产出”里显式写出来

### 2. `projects/` 适用

适合个人项目、课程项目、工程项目、开源项目。通常建议补这些字段：

```md
- 项目目录：
  - 
- 相关仓库：
  - 
```

写法重点：

- 优先说明你主导了什么，而不是泛泛写“参与开发”
- 可以把“主要贡献”拆成 numbered sections，比如 Runtime、MCP、Parsing、Testing
- 对 agent / system / infra 类项目，建议明确：
  - 输入输出
  - 核心模块
  - 安全边界
  - 测试或验证方式

### 3. `publications/` 适用

适合论文条目。通常建议补这些字段：

```md
- 会议 / 期刊：
- 作者：
- 机构：
- arXiv：
- 项目主页：
- DOI：
- 论文解析：
```

写法重点：

- 不只写 bibliographic info，要写这篇论文的研究价值和关键设定
- 如果和你自己的经历有关，单独加一节“与我的经历的关联”
- 如果库里有详细解析，链接到 `publications/papers/` 对应文件

---

## 写作提示

- “一句话概述”更像你以后在简历或面试里的总起句
- “我的工作”要比简历更细，后续才能自由裁剪
- “结果与产出”不要只写“效果很好”，尽量落到数字、系统能力或正式产出
- “后续可补充”是刻意保留的不确定项，不是缺点

## 最小合格标准

一个新条目至少应包含：

1. `基本信息`
2. `一句话概述`
3. `背景与目标`
4. `我的工作`
5. `技术方案`
6. `结果与产出`

如果暂时信息不全，也至少把这 6 部分先搭起来。
